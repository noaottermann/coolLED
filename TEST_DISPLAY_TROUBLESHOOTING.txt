If test_display.py does not work, check the following:

1. Make sure the dependencies are installed.
   Run: pip install -r requirements.txt

2. Confirm Bluetooth is enabled and the CoolLED device is not already connected to another app.
   Close any phone app or other program that may be using the sign.

3. Verify the MAC address in utils/config.py.
   If your device address is different, update MAC_ADDRESS before running the script.

4. Confirm the display size matches your device.
   If the sign is not 64x16, update DISPLAY_WIDTH and DISPLAY_HEIGHT in utils/config.py.

5. Run the script from the repository root.
   Example: python test_display.py

6. If the script connects but no pixels appear:
   - Try lowering or raising the brightness value in test_display.py.
   - Check that the device is powered on.
   - Make sure the sign is actually the model you expect.
   - Try a very simple test pattern first, such as a single bright pixel or a full solid color.

7. If the script fails to connect:
   - Move closer to the device.
   - Turn Bluetooth off and back on.
   - Restart the sign and your PC Bluetooth adapter.
   - Wait a few seconds after connecting; some devices need a short delay before they accept image data.

8. If you get an import error or module error:
   - Run the script from the project root folder.
   - Confirm you are using the correct Python environment.

9. If the script says it sent data but the LEDs stay blank:
   - Double-check that the sign dimensions are correct.
   - Try another known-working command or a different test image.
   - Reboot the display and try again.

If none of this helps, the next useful step is to capture the exact error message and check whether the sign is advertising the correct CoolLED model and firmware behavior.

If a test fails while you are running pytest, use the error message to choose the next step:

1. If you get an ImportError or ModuleNotFoundError:
   - Run the tests from the repository root.
   - Make sure the project root is on the Python path.
   - Check that the module name in the import matches the file you created.

2. If you get a SyntaxError or IndentationError:
   - Open the file named in the traceback.
   - Check for a missing colon, broken indentation, or an unfinished block.
   - Run the same test again after fixing the syntax.

3. If you get a FileNotFoundError:
   - Check whether the test expects a file from the tests folder.
   - Make sure any sample image, animation, or JT file really exists.
   - Confirm that the test is using the right relative path.

4. If you get a failure about an expected command payload:
   - The rendering code probably changed.
   - Re-run the command-generation tests and compare the new payloads.
   - Update the expected test data only after you are sure the new output is correct.

5. If you get a Bluetooth or Bleak error during a test:
   - Check whether the test actually needs the hardware connected.
   - If it does, make sure the device is powered on and not connected elsewhere.
   - If the test is supposed to be offline, mock or skip the BLE part.

6. If you get a TimeoutError:
   - Try again once to see whether the device was just slow to respond.
   - Increase the timeout if the code has a configurable timeout value.
   - Make sure the panel is awake and reachable before the test starts.

7. If a test passes locally but fails on another machine:
   - Check the Python version.
   - Compare installed dependencies.
   - Look for Windows-specific path or Bluetooth behavior.

When you do not know what to fix, copy the full traceback and match the topmost failing line to the closest file in this repository. That is usually the fastest way to find the real problem.
